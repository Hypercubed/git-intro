# Why git?
## Git vs. Whatever - Not better, different - No server required, Off-line - Tracks content not files - Branches are lightweight

Git workflow

  1. (git init)
  2. Edit files
  3. Review changes (git status)
  4. Stage changes (git add)
  5. Commit changes (git commit)
This is a basic git workflow. After making changes to files, you review those changes, stage changes, and commit. (space) These map to the three fundamental git commands that we'll discuss today. Plus one more.
# git init #### Initialize a new Git repository - Create a new empty repository in the current folder - Does not changes any files you have already created ```(bash) > cd mywork > git init Initialized empty Git repository in /Scratch/mywork/.git/ > ll -a .git/ data/ mydoc.txt scripts/ ``` Note: git init a new empty repository in the current folder. If you see here I'm c-d-ed into an exsiting project. Git init creates a single hidden `.git` directory. My existing files are not touched at all.
#### (2. Review changes) # git status ### ```(bash) > git status # On branch master # # Initial commit # # Untracked files: # (use "git add ..." to include in what will be committed) # # data/ # mydoc.txt # scripts/ nothing added to commit but untracked files present (use "git add" to track) ``` Note: git status shows a status for teh current git repository. You can see that I have one file and two fodlers in this directory that are not currently tracked.
#### (2. Review changes) # git diff ### ```(bash) > git diff diff --git a/scripts/myscript.sh b/scripts/myscript.sh index 9daeafb..038d718 100644 --- a/scripts/myscript.sh +++ b/scripts/myscript.sh @@ -1 +1 @@ -test +testing ``` Note: git diff is not one of the core commands I mentioned but it helpful for seeing exactly what changes were made.
#### (3. Stage changes) # git add ### - Doesn't alter files - Add only what you want ```(bash) > git add mydoc.txt scripts/ > git status # On branch master # # Initial commit # # Changes to be committed: # (use "git rm --cached ..." to unstage) # # new file: mydoc.txt # new file: scripts/myscript.sh # # Untracked files: # (use "git add ..." to include in what will be committed) # # data/ ``` Note: `git add` tells git that you want to include the current version of a file in the next commit. It doesn't alter the file at all. Later when you make a change to a file you run `git add` again to say you want those changes. Notice that I didn't add my data/ folder. You don't need to add everything.
#### (4. Commit changes) # git log #### a journal of changes ``` > git log commit 8e2bf4a56ed63c8921ccffa751f6c1783f4b542e Author: J. Harshbarger Date: Tue Jun 10 14:32:48 2014 +0900 Second commit 5053e8cbed15e7e2323fcb15b022847db9863526 Author: J. Harshbarger Date: Tue Jun 10 14:25:55 2014 +0900 First commit ``` Note: Another useful command not in the core commands is git log.
# That's the basics Notes: And that is all you really need to get started using git on your local machine.
#### (branching) # git merge #### Take changes from one branch and merge them with a second. ``` > git checkout master Switched to branch 'master' > git merge test Updating 5053e8c..8e2bf4a Fast-forward mydoc.txt | 1 + scripts/myscript.sh | 1 + 2 files changed, 2 insertions(+), 0 deletions(-) ``` Note: After I've create a new branch, switches to it and committed changes (remember git add and git commit?) I can switch back to another branch and merge changes. When you do the `git checkout` the files in your file system will change from they way they were in one branch to the other. Now git is making changes to your files!
# GIT remotes #### GIT remotes are remote locations where you can mirror changes - Online (GitHub/Bitbucket) - Another machine in your network - Over ssh ``` > git init --bare //osc-fs/jayson/repos/example.git Initialized empty Git repository in //osc-fs/jayson/repos/example.git/ > git remote add oscfs //osc-fs/jayson/repos/example.git > git push oscfs master Counting objects: 8, done. Delta compression using up to 4 threads. Compressing objects: 100% (4/4), done. Writing objects: 100% (8/8), 609 bytes, done. Total 8 (delta 0), reused 0 (delta 0) Unpacking objects: 100% (8/8), done. To //osc-fs/jayson/repos/example.git * [new branch] master -> master ``` Note: `git remote` is how you can share or backup your repo to another location. You can push your code to github or bitbucket for example or to another machine.... even over ssh. You see here I've created a new git repository (in this case it is a special bare repository) on osc-fs. I then added this location to my local repository calling it oscfs. Then I can easily copy branches from my local machine to osc-fs.
